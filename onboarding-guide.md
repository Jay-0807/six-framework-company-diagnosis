# Onboarding:首次调用引导

**触发条件**:使用者第一次调用 skill 时,`~/.company-diagnosis/user-profile.md` 不存在。

**绝对规则**:在 onboarding 完成前 **禁止开始任何六框架诊断**。退回引导。

---

## Step 1:解释这个 skill 需要什么

跟使用者说(中文,简短):

> 这个 skill 给你做企业诊断时会锚定到 **你自己的商业路径**。我先问你 4 块基本信息,写到本地 `~/.company-diagnosis/user-profile.md`。这个文件会随你不断调研和自我迭代而更新,不会进任何公开仓库。
>
> 全部填完大约需要 10-15 分钟。中途可以跳过非必填项。

## Step 2:用 AskUserQuestion 分块收集

按 [user-profile-template.md](user-profile-template.md) 的字段,分 5 块(Block 0 + Block 1-4)收集。

> **顺序铁律**:Block 0 必须最先问,因为它决定了第七节"对使用者商业路径的启示"该回答什么具体问题。

### Block 0:本次诊断目的(必填,新增 v3)

**问 1 件事**(可选项 + Other):

- 你这次要做诊断,**最想搞清楚什么具体问题**?(可以列 1-3 个,目的越具体,第七节回答得越精准)

可选项参考(让使用者快速选 1 个,然后允许补充):
- 我想看这家公司的护城河是什么 / 我能不能学
- 我想看这家公司怎么做客户细分 / 人群切割
- 我想看这家公司怎么做组织(管理 / 激励 / 中台)
- 我想看这家公司的扩张路径(品类 / 渠道 / 地区 / 多品牌)
- 我想看这家公司有什么失误 / 风险 / 我应该规避什么
- 我想做收购对照 / 投资判断
- 其他(自填)

❌ 不要接受"看看而已 / 学习一下"——逼使用者落到 **"我想搞清楚什么"**。这是后面第七节锚定的依据。

### Block 1:身份定位(必填,**v3 改造**)

> **v3 改造重点**:行业先行,商业模式自适应——避免默认 SaaS / 互联网偏见。

#### Block 1a:先问行业

**问 1 件事**:你公司**所属行业**是?

可选项(覆盖典型行业;使用者从中选 1 个,允许 Other):
- 电商 / 零售 / 消费品(含代运营、自有品牌、线上线下零售)
- SaaS / 软件 / IT 服务
- 互联网 / 平台经济 / 内容
- 制造业 / 工业 / 硬件
- 服务业(咨询 / 设计 / MCN / 教育 / 医疗 / 法律等)
- 金融 / 投资 / 资管
- 文创 / 媒体 / 娱乐
- 农业 / 食品 / 餐饮
- 房产 / 建筑 / 物业
- 跨境 / 外贸 / 国际化业务
- 其他(自填)

#### Block 1b:根据行业自适应问商业模式

LLM 必须根据使用者 Block 1a 的选择,**动态给出对应行业的商业模式选项**。下表是参考映射(允许 Other 自填):

| 行业 | 常见商业模式选项 |
|---|---|
| **电商 / 零售 / 消费品** | ① 自有品牌卖货毛利 ② 代运营卖货毛利(品牌授权 + 自营供应链) ③ 代运营服务费 ④ 平台 TP / 抽佣 ⑤ 经销 / 分销 ⑥ 私域 / 订阅 ⑦ 直播带货 / 内容电商 |
| **SaaS / 软件** | ① 订阅 SaaS ② 项目制 + 实施费 ③ License 一次性 ④ PaaS / API 用量 ⑤ 混合(订阅 + 增值服务) |
| **互联网 / 平台** | ① 广告 ② 抽佣 ③ 会员订阅 ④ 增值服务 ⑤ 流量分发 |
| **制造业** | ① 直供品牌方 ② OEM/ODM 代工 ③ 自有品牌 ④ 经销 |
| **服务业** | ① 项目制咨询费 ② 顾问费 / 时薪 ③ 订阅服务 ④ 包年包月 ⑤ 抽佣 / 分成 |
| **金融** | ① 利差 / 息差 ② 管理费 ③ 抽佣 / 业绩报酬 ④ 牌照租赁 |
| **文创 / 媒体** | ① 广告 ② 内容订阅 / 付费 ③ IP 授权 ④ 出版 / 发行 ⑤ 制作分成 |
| **农业 / 食品** | ① 自营卖货 ② 加工出售 ③ 农资 / 服务 ④ 订单农业 |
| **跨境 / 外贸** | ① 卖货毛利 ② 服务费 ③ 一件代发 ④ DTC 自有品牌 ⑤ B2B 大宗贸易 |
| **其他 / 自填** | 让使用者直接描述 |

❌ 不要接受"卖产品 / 提供服务"这种泛泛回答——逼使用者落到"**卖谁、收谁的钱、怎么收**"。

#### Block 1c:一句话业务

**问 1 件事**:用**不超过 25 字**描述你公司的一句话业务。

❌ 不要接受这种:"我们是做 AI 创新的"——要求**让陌生人立刻明白你做什么**,要具体到品类 + 客户 + 模式。

✅ 例:"抖音/视频号女性内衣品类代运营,授权品牌贴牌+自营供应链"

### Block 2:客户与阶段(必填)

**问 2 件事**:
- 目标客户画像 = 行业 + 规模 + 决策角色(例:"年营收 3-30 亿的传统制造业老板",或 C 端则给"性别 + 年龄段 + 价位带 + 场景")
- 当前阶段 = `创意 / MVP / 早期 PMF / 规模化 / 成熟期` 五选一 + 关键证据("已有 N 个付费客户" / "MRR 多少" / "GMV 多少" / "团队规模")

### Block 3:核心商业纪律(必填,3-5 条;**v3 改造**)

> **v3 改造重点**:允许使用者跳过直接填,改为 skill 根据 Block 1-2 信息**自动推导候选红线**,使用者只需 yes/no 确认。

**理想情况**:使用者直接给"合同条款级"的具体表述。

❌ 反例:「我们坚持高品质交付」「以客户为中心」「长期主义」
✅ 正例:
- 「首单必须纯咨询,不打包咨询 + 软件混合单」
- 「不接年营收 < 3000 万的客户」
- 「单客户咨询费 ≥ 15 万元起」
- 「不上架钉钉 / 飞书 / 企微的服务商市场」

**如果使用者填不出**(常见情况):

进入 **"自动推导 + 确认"模式**:
1. LLM 根据 Block 1-2 的行业、商业模式、客户画像、阶段,**推导 3-5 条候选红线**写在 user-profile.md 的"候选红线"段
2. 候选红线**必须标 `[v0.5-推导]`**(版本号),并明确告诉使用者"这是 skill 推导出来的,不是你拍板的"
3. 在第一次诊断结束的第七节 7.3 中,用诊断结论挑战 / 验证 / 修订这些候选红线,**让使用者拍板**
4. 拍板后红线升级为 `[v1-用户确认]`,profile 版本号从 v0.5 → v1.0

**自动推导参考逻辑**:

| 业务类型线索 | 候选红线模板 |
|---|---|
| 代运营/卖货毛利 | 「不做纯白牌,必须有品牌冠名」「主战场聚焦 X 渠道」「品类聚焦在 X」 |
| SaaS 订阅 | 「客单价 ≥ X」「不接低于 N 天 POC 的客户」「优先大客户/标杆 + SMB 自服务」 |
| 项目制咨询 | 「单客户咨询费 ≥ X 万起」「不打包软件 / 不抽佣」「客户必须是创始人决策」 |
| 制造业 | 「不接低于 X 万的订单」「不做账期超过 N 天的客户」「核心工艺保留自营」 |
| 内容/媒体 | 「不接违背价值观的广告」「订阅模式优先于广告模式」 |

### Block 4:路线图与不确定性(推荐填,但允许暂缺)

**问 3 件事**:
- 重点对标 / 参照公司:2-5 家 + 至少 1 家标记为"必看基准"
- 当前 12 个月路线图:3-5 个关键里程碑 + 时间窗(按季度)
- 关键不确定性:3-5 个让你睡不着的问题(**强烈建议至少填 1 条**——第七节回答它)

这块允许"先填部分",但要标 `[v0-暂缺]`,之后慢慢补。

## Step 3:写入文件

按 [user-profile-template.md](user-profile-template.md) 的格式,把 5 块信息组装成完整 markdown,写入:

```
~/.company-diagnosis/user-profile.md
```

第一行版本号:
- Block 0-3 都齐(包括 v3 推导红线已经用户确认)→ `v1.0`
- Block 0-3 齐但红线是推导版(未确认)→ `v0.5`
- 跳过 Block 3 红线 → `v0-skipped-block3`

最后更新日期 = 当天。

## Step 4:确认 + 进入诊断

跟使用者复述一遍 5 块要点,**特别强调 Block 0 本次诊断目的**(因为第七节会专门回答),确认无误后开始六框架分析。

---

## 跳过 / 简化模式

如果使用者明确说"不要 onboarding,直接分析":

1. 创建一个空壳 profile:
   ```markdown
   # 未命名使用者 商业路径规划(精简模式)

   **版本**:v0-skipped
   **最后更新**:YYYY-MM-DD

   ⚠️ 使用者跳过了 onboarding。第七节"对使用者商业路径的启示"将按通用模式输出,无法做字段级反哺建议。
   ```

2. 在最终报告**第七节顶部加红字提示**:
   > ⚠️ 使用者跳过了 onboarding,本节按通用模式输出。强烈建议补 onboarding 后重跑以获得字段级反哺建议。

3. 不要因为 skipped 就放弃归档到 `memory/`——后期补 onboarding 后这些历史调研也有意义。

---

## 后续迭代

`user-profile.md` 不是一次填完。**每次六框架诊断结束后第七节会给字段级更新建议**。使用者拍板后:
- 旧版本归档到 `~/.company-diagnosis/user-profile.history/<旧版本号>-<日期>.md`
- 新版本号 v_._ → v_._+1
- 顶部更新 "最后更新" 日期

记忆系统详见 [memory-spec.md](memory-spec.md)。
